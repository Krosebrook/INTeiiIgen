## ğŸ“‹ Phase 1: Planning (ALWAYS START HERE)

### When to Use
- New project
- New feature (even "small" ones)
- Unclear requirements
- High-risk changes

### Prompts (In Order)

**1.1 Requirements Gathering**
- Input: Problem statement, stakeholders, constraints
- Output: Clear requirements doc (1-2 pages)
- Next: Architecture Design

**1.2 Architecture Design**
- Input: Requirements from 1.1
- Output: System diagram, tech stack, data model
- Next: Effort Estimation

**1.3 Effort Estimation**
- Input: Architecture from 1.2
- Output: Time estimate (best/likely/worst case)
- Next: Technical Feasibility

**1.4 Technical Feasibility**
- Input: Architecture + Estimates
- Output: Go/no-go decision, risks, mitigations
- Next: Phase 2 (Design) OR Phase 3 (Development)

### Exit Criteria
- [ ] Requirements documented and reviewed
- [ ] Architecture diagram exists
- [ ] Effort estimate has confidence level
- [ ] No technical blockers

### Common Mistakes
âŒ Skipping this phase ("I already know what I need")  
âŒ Verbal requirements (not written)  
âŒ No architecture diagram  
âŒ Guess estimates ("probably 2 weeks")

### Fast-Track Exceptions
- Bug fixes (requirements = bug report)
- Hotfixes (architecture = existing system)
- Refactors (requirements = improve code quality)
